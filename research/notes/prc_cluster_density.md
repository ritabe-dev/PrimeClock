# PRC Cluster Density D_R

This note records the first dedicated `D_R` view.

## Definition

For a radius `R`, define local complete-covering density by:

```text
D_R(N) = #{M in [N-R,N+R] : C0(M)=1} / (2R+1)
```

where `C0(M)=1` means exact complete covering:

```text
A(M) = |T \ U_M| = 0
```

In computation, `C0(M)=1` is only counted after exact interval certification.
The scanner checks every integer in the selected local window, so `D_R` is not
a float-zero lower bound.

## Reading

`D_R` is better than just listing complete-covering values because it turns
isolated-looking hits into a local statistical object. The object is still
finite and empirical, but it lets us ask sharper questions:

```text
Where are exact complete-covering values locally dense?
How stable is that density under changing R?
Are dense windows simply low-A windows, or something more arithmetic?
```

The first `D_250` scan suggests that selected efficient windows are neither
consecutive blocks nor single isolated hits. Complete-covering values are
sprinkled through these windows at roughly `2%` to `3.4%` density. The median
window still has substantial uncovered mass relative to the random baseline, so
a dense cluster is not a whole neighborhood where every `N` almost covers the
circle.

This is important: the local environment can look ordinary by median `A(N)`,
while still containing many exact zeros. That makes `D_R` a more interesting
target than a single `C0(N)` list. The result is still seed-conditioned: the
windows were selected by a coarse covering-efficiency scan, then analyzed for
local complete-covering density.

## Next Checks

- Sensitivity to `R`: compare `R=100`, `250`, and `500`.
- Sensitivity to the coarse seed threshold: compare exact-zero seeds only with
  `A/baseline <= 0.02` and `A/baseline <= 0.05`.
- De-duplicate overlapping windows when reporting global counts.
- Add a null model for the number of exact hits per window. The current random
  arc baseline controls `A(N)` scale, but it does not yet predict `C0(N)`.

## Sensitivity v0

The sensitivity table is generated by:

```bash
python -m prime_reciprocal_projection.cli covering-cluster-sensitivity \
  --start 1000 \
  --stop 1000000 \
  --count 500 \
  --ratio-threshold 0 0.02 0.05 \
  --radius 100 250 500 \
  --out data/summaries/prc_cluster_sensitivity_v0.csv
```

Output:

```text
data/summaries/prc_cluster_sensitivity_v0.csv
```

Reading:

- `threshold=0` and `threshold=0.02` are identical on this scan. The coarse
  grid contributed exact-zero seeds, but no additional seeds with
  `0 < A/baseline <= 0.02`.
- Adding `threshold=0.05` increases the number of seed windows from `12` to
  `21`, and increases the total unique exact values found. But the mean density
  slightly decreases, because the added windows are weaker on average.
- On these selected seed sets, the center-weighted mean `D_R` stays near
  `0.024` to `0.027` when `R` changes from `100` to `500`.
- The top center changes with `R`. This means "best cluster" is scale-dependent:
  `N=51692` is strongest for `R=100/250`, while wider windows favor `N=58551`
  or `N=3524`.

This supports using `D_R` as a local-density observable for selected windows,
but also warns against naming a single universal "best" cluster without
specifying `R`, the seed rule, and whether the summary is center-weighted or
unique-value based.
